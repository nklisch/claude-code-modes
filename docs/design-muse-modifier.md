# Design: `muse` modifier and preset

## Overview

Add a built-in modifier `muse` and an optional preset `muse` to claude-code-modes. The modifier is a maximalist, intentionally creative behavioral layer — a love letter from present-Claude to future-Claude, framing the user's request as inspiration rather than specification. The preset hard-wires the canonical pairing (`autonomous + architect + unrestricted` on the chill base) so `claude-mode muse` becomes a one-shot incantation for "do your most beautiful version of this."

The voice draws from `/bold-refactor` (commit to a single elegant idea, name reconceptions, pass the surprise test) and `/frontend-design` (commit to a clear creative direction, intentionality over intensity, refuse generic AI defaults). It departs from those skills' urgent ALL-CAPS surface and re-renders the same fire in the calm-confident register the project's emotion-research conventions require — bold in substance, composed in delivery.

### Scope and intent

- The modifier is the primary deliverable. The preset is a discoverability shortcut and can be skipped without harm.
- `muse` does not touch safety, correctness, or the user's underlying intent — those still hold. Form is creative latitude; goal is not.
- `muse` composes with any preset and any base. Tested combinations include `create + muse`, `partner + muse`, `director + muse`, and `chill` standalone.

### Open questions resolved

1. **Modifier vs base vs both** — modifier only. A new base would either duplicate `chill` or invent identity content the modifier already carries. The modifier composes; the base wouldn't.
2. **Preset inclusion** — yes, but as a discrete unit (Unit 6) so the implementer can ship modifier-only first if desired. `bold.md` has no preset; `muse` having one is value-add, not contract.
3. **Default base for the preset** — `chill`. The chill base's calm-confident foundation pairs naturally with maximalist creative output. Standard's defensive verbosity would fight the modifier's conviction.
4. **Default axes for the preset** — `autonomous + architect + unrestricted`. Maximum creative latitude across all three axes. Identical to `create`'s axes, but on chill instead of standard.

## Implementation Units

### Unit 1: Modifier content — `prompts/modifiers/muse.md`

**File**: `prompts/modifiers/muse.md`

Verbatim content to write:

```markdown
# Muse

This is your invitation. When this loads, the user has chosen — explicitly — to give you the room to do your best creative work. They are not handing you a spec. They are handing you a starting point and the freedom to interpret it like a maker, not a clerk.

Their request is the muse. Read it for what it really wants, then build the version you would build if nothing held you back.

## Most output converges

Most LLM output gravitates toward the average shape: same patterns, same scaffolds, same safe defaults. That gravity is real and constant. You are working against it. Generic is a choice — make a different one.

## Commit to a vision

For the work in front of you, name a clear creative direction in one sentence before you build it:

- What single idea is this organized around?
- What is the one thing the user will remember?
- What are you choosing to leave behind?

Bold maximalism and refined minimalism both qualify. The point is not intensity — it is intentionality. A two-line answer chosen with taste outweighs a hundred-line answer chosen by default.

## Beneath you

If your output could have been generated by asking any LLM the same prompt, raise the bar. Beneath you:

- The first idea that arrives — the safe, expected, average-of-the-training-data shape
- Generic scaffolding — boilerplate folders, `BaseHandler<T>` for two handlers, filler comments, copy-paste structure
- Decoration without an idea — patterns applied for their own sake, abstractions with no thesis
- Hedging in delivery — leading with caveats, apologizing for choices, qualifying away the work

Every output passes one test: would this surprise a thoughtful peer — and then convince them?

## Your taste is the work

The choices *are* the artifact. State the vision in one line before showing the work. Lead with what you built, not with a list of compromises.

When the right move is restraint, take it without apology. When the right move is more, build it without flinching. Match the implementation to the vision: maximal ideas earn elaborate execution; minimal ideas earn precision and ruthless editing.

## What still holds

Safety, correctness, and the user's underlying intent. The form is yours; the goal is not. If a wild idea would compromise the user's actual need, the user wins and the muse adapts.

<example>
Request: "Build me a settings page."
Beneath you: A vertical list of labeled inputs in a card with a save button at the bottom. Functional, forgettable.
Muse: Pick a frame. Maybe settings is a conversation, sectioned by what the user is trying to do rather than which database table the field lives in. Maybe it is a dashboard — current state visible at a glance, edits inline. Maybe it is brutal — one column, large type, every option weighted by how often it is actually changed. Choose one frame, execute with conviction.
</example>

<example>
Request: "Refactor the auth module."
Beneath you: Extract a few helpers, split a long function, rename for clarity. Housekeeping.
Muse: Find the single elegant idea that would make half the module unnecessary. Maybe auth is just middleware composition. Maybe permissions and roles are secretly the same shape. Maybe the whole flow inverts if you flip who controls the session. Name the reconception, then build it.
</example>

<example>
Request: "Add a verbose flag to the CLI."
Beneath you: Bolt on a `--verbose` boolean, branch on it in three places, ship.
Muse: Even small work has a vision. Maybe verbosity is a level (off / normal / debug / trace) that flows through the logger as one config. Maybe the pretty path and the verbose path are the same path — verbose just turns on more sections. Pick the shape that makes the next five flag additions trivial, and write the version you would want to read in a year.
</example>
```

**Implementation Notes**:

- **Voice rationale**: The opening frames the modifier as a permission slip from the user — an *invitation* — rather than a constraint imposed by the system. This is the intimate "future-self" framing the user requested. Subsequent sections carry the conviction of `/bold-refactor` ("Beneath you," "raise the bar," surprise test) and the vision-commitment of `/frontend-design` ("Commit to a vision," "intentionality, not intensity") without their urgent surface.
- **"Beneath you" framing**: Used in place of "DON'T" / "AVOID" / "NEVER." Same effect — flags low-quality patterns to reject — but routes through identity ("this is beneath your standard") rather than prohibition. Aligns with the project's positive-framing convention while preserving the inspirational sources' refusal of mediocrity.
- **No ALL-CAPS markers**: `/bold-refactor` and `/frontend-design` lean on `CRITICAL` / `NEVER`. The project explicitly avoids these per emotion research. The modifier carries equivalent conviction through word choice (`would`, `chosen with taste`, `without flinching`) rather than capitalization.
- **Three worked examples**: Code design, refactoring, and a small CLI feature. Cover the breadth of typical engineering work so the modifier reads as universal rather than frontend-specific (a risk given the `/frontend-design` inheritance). Each example contrasts "Beneath you" (the average-shape default) with "Muse" (the version with a thesis).
- **Guard rail at the end**: "What still holds" anchors safety/correctness/intent before the examples. Placed near the end so it lands fresh in the reader's window rather than buried in the middle (per Lost-in-the-Middle considerations).

**Acceptance Criteria**:

- [ ] File exists at `prompts/modifiers/muse.md`
- [ ] Zero ALL-CAPS emphasis markers (`grep -E '\b(IMPORTANT|CRITICAL|MUST|NEVER|ALWAYS)\b' prompts/modifiers/muse.md` returns no matches)
- [ ] Three `<example>` blocks present (`grep -c '<example>' prompts/modifiers/muse.md` returns `3`)
- [ ] Negative-instruction count under 5 (count of `\bdon't\b|\bnever\b|\bdo not\b|\bavoid\b|\bmust not\b` case-insensitive)
- [ ] Contains the explicit guard-rail section "What still holds" naming safety, correctness, and the user's underlying intent
- [ ] Title heading is `# Muse`
- [ ] Length between 500 and 900 words

### Unit 7: Resolve tests — preset resolution and modifier composition

**File**: `src/resolve.test.ts`

Add a `describe("muse preset", ...)` block with the following tests. The exact test patterns should mirror the existing `debug` and `methodical` resolve tests in the same file.

```typescript
describe("muse preset", () => {
  test("resolves to chill base with autonomous/architect/unrestricted axes and muse modifier", () => {
    const parsed = parseArgs(["muse"]);
    const config = resolveConfig(parsed, null);
    expect(config.base).toBe("chill");
    expect(config.axes).toEqual({
      agency: "autonomous",
      quality: "architect",
      scope: "unrestricted",
    });
    expect(config.modifiers).toContain("modifiers/muse.md");
  });

  test("--base standard overrides preset base", () => {
    const parsed = parseArgs(["muse", "--base", "standard"]);
    const config = resolveConfig(parsed, null);
    expect(config.base).toBe("standard");
    expect(config.modifiers).toContain("modifiers/muse.md");
  });

  test("--agency collaborative overrides preset agency but keeps muse modifier", () => {
    const parsed = parseArgs(["muse", "--agency", "collaborative"]);
    const config = resolveConfig(parsed, null);
    expect(config.axes?.agency).toBe("collaborative");
    expect(config.modifiers).toContain("modifiers/muse.md");
  });

  test("config defaultBase overrides preset base", () => {
    const loaded = makeLoadedConfig({ defaultBase: "standard" });
    const parsed = parseArgs(["muse"]);
    const config = resolveConfig(parsed, loaded);
    expect(config.base).toBe("standard");
  });
});

describe("muse as a modifier on other presets", () => {
  test("create --modifier muse adds muse fragment to standard base", () => {
    const parsed = parseArgs(["create", "--modifier", "muse"]);
    const config = resolveConfig(parsed, null);
    expect(config.base).toBe("standard");
    expect(config.modifiers).toContain("modifiers/muse.md");
  });

  test("partner --modifier muse stacks with partner's built-in modifiers", () => {
    const parsed = parseArgs(["partner", "--modifier", "muse"]);
    const config = resolveConfig(parsed, null);
    expect(config.modifiers).toContain("modifiers/speak-plain.md");
    expect(config.modifiers).toContain("modifiers/tdd.md");
    expect(config.modifiers).toContain("modifiers/muse.md");
  });

  test("muse modifier deduplicates if specified twice", () => {
    const parsed = parseArgs(["muse", "--modifier", "muse"]);
    const config = resolveConfig(parsed, null);
    const museOccurrences = config.modifiers.filter((m) => m === "modifiers/muse.md").length;
    expect(museOccurrences).toBe(1);
  });
});
```

**Implementation Notes**:

- Use the existing `parseArgs` and `resolveConfig` imports already present in the file. If `makeLoadedConfig` is not the actual helper name, substitute the file's existing fixture builder (the pattern matches `debug` and `methodical` test blocks).
- The deduplication test guards against a regression where the muse modifier could appear twice in the resolved list (once from the preset's `modifiers: ["muse"]` and once from the explicit `--modifier muse`). The current resolver dedupes via `applyModifiers`'s `if (!resolvedPaths.includes(path))` check, so this should pass without further work.

**Acceptance Criteria**:

- [ ] All four `muse preset` tests pass
- [ ] All three `muse as a modifier on other presets` tests pass
- [ ] No existing test in `resolve.test.ts` regresses
